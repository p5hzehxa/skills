## Step 4: Webhook Implementation (If Real-Time Push Required)

### 4A: Create Webhook Endpoint

**Framework-specific patterns:**

```typescript
// Express.js pattern (Node.js)
app.post('/webhooks/workos', 
  express.raw({ type: 'application/json' }), // CRITICAL: raw body needed for signature
  async (req, res) => {
    // 1. Verify signature FIRST (Step 4B)
    // 2. Return 200 immediately
    // 3. Process async (background job)
  }
);

// Next.js App Router pattern
export async function POST(request: Request) {
  const body = await request.text(); // raw text for signature
  // 1. Verify signature
  // 2. Return Response with 200
  // 3. Process async
}
```

**Flask pattern (Python):**
```python
@app.route('/webhooks/workos', methods=['POST'])
def workos_webhook():
    payload = request.data  # raw bytes for signature
    # 1. Verify signature
    # 2. Return 200
    # 3. Process async
```

**Key requirements:**
- Accept POST requests
- Read raw request body (NOT parsed JSON) for signature verification
- Return HTTP 200 immediately (within 5 seconds)
- Process events asynchronously after responding

**Why raw body:** Signature is computed over exact bytes. Parsed JSON won't match.

### 4B: Signature Verification (MANDATORY)

**Never process webhook payload without verification.** Unverified webhooks = security vulnerability.

```
Verification flow:
  |
  +-- Extract header --> Parse timestamp (t=) and signature (v1=)
  |
  +-- Check timestamp --> If >5 minutes old, reject (replay attack)
  |
  +-- Compute expected --> HMAC-SHA256(webhook_secret, timestamp.payload)
  |
  +-- Compare --> If mismatch, return 401
  |
  +-- Match --> Safe to process
```

**SDK method pattern (preferred):**

```python
# Python example - check fetched docs for exact signature
from workos import WebhookClient

webhook = WebhookClient(webhook_secret)
try:
    event = webhook.verify_event(
        payload=request.data,
        sig_header=request.headers.get('WorkOS-Signature'),
        tolerance=300  # 5 minutes
    )
    # event is verified, safe to process
except Exception:
    return jsonify({'error': 'Invalid signature'}), 401
```

**Manual verification pattern (if SDK unavailable):**

1. Parse `WorkOS-Signature` header: split on `,` to get `t=` and `v1=` values
2. Extract timestamp from `t=<milliseconds>`
3. Reject if `current_time - timestamp > 300000` milliseconds
4. Construct signature payload: `{timestamp}.{raw_request_body}`
5. Compute HMAC-SHA256 with webhook secret
6. Compare computed signature to extracted `v1=` value (constant-time comparison)

### 4D: Response Pattern (CRITICAL)

**Anti-pattern (WRONG):**
```javascript
// DON'T DO THIS - blocks response
app.post('/webhooks/workos', async (req, res) => {
  verifySignature(req);
  await processEvent(req.body); // long operation
  res.sendStatus(200); // timeout if >30 seconds
});
```

**Correct pattern:**
```javascript
app.post('/webhooks/workos', async (req, res) => {
  verifySignature(req);
  res.sendStatus(200); // respond FIRST
  
  // Queue for async processing
  await queue.add('workos-event', req.body);
});
```

**Why:** WorkOS expects 200 within seconds. Long processing triggers retries.

**Retry behavior:**
- WorkOS retries failed deliveries 6 times
- Exponential backoff over 3 days
- Each retry has same signature (idempotency key = event ID)

**Idempotency handling:**
```
On event receipt:
  |
  +-- Check if event.id exists in processed_events table
  |     |
  |     +-- EXISTS --> Return 200, do nothing (already processed)
  |     +-- NEW --> Process + store event.id
```

## Step 6: Events API (If Polling/Batch Required)

**Use case:** Reconciliation, backfill, or systems that can't receive webhooks.

### 6A: List Events Endpoint Pattern

```
GET /events?after={cursor}&limit={count}
Authorization: Bearer {WORKOS_API_KEY}
```

**Pagination pattern:**
```python
# Pseudocode - check fetched docs for exact SDK method
cursor = None
while True:
    response = workos_client.list_events(after=cursor, limit=100)
    for event in response.data:
        process(event)
    
    if not response.has_more:
        break
    cursor = response.data[-1].id  # last event ID becomes next cursor
```

**Critical:** Events API does NOT guarantee ordering. For strict ordering, use webhooks.

### 6B: Cursor-Based Pagination

- `after` parameter: Event ID to start after (exclusive)
- `limit` parameter: Max events per page (typically 100)
- `has_more` in response: Boolean indicating more pages exist

**Store cursor:** Persist last processed event ID to resume from crashes.

## Step 7: Event Processing Patterns

### 7A: Event Structure

All events share this structure:
```json
{
  "id": "event_01H...",
  "event": "directory.user.created",
  "data": { /* event-specific payload */ },
  "created_at": "2024-01-15T10:30:00.000Z"
}
```

**Unique event identifier:** `event.id` (use for idempotency)
**Event type:** `event.event` (use for routing)

### 7B: Event Routing Pattern

```javascript
// Event handler router
function handleEvent(event) {
  switch(event.event) {
    case 'directory.user.created':
      return syncUserToDatabase(event.data);
    
    case 'directory.user.updated':
      return updateUserInDatabase(event.data);
    
    case 'authentication.email_verification_succeeded':
      return markEmailVerified(event.data);
    
    default:
      console.warn(`Unhandled event type: ${event.event}`);
  }
}
```

**Trap:** New event types may be added. Log unknowns instead of erroring.

## Error Recovery

### "Invalid signature" on webhook

**Root cause:** Signature mismatch between WorkOS and your verification.

Fixes:
1. Verify `WORKOS_WEBHOOK_SECRET` matches Dashboard value (copy again)
2. Confirm using RAW request body for verification (not parsed JSON)
3. Check timestamp tolerance isn't too strict (<5 minutes recommended)
4. Verify HMAC algorithm is SHA256 (not SHA1/MD5)

**Test command:**
```bash
# Check if raw body preserved
curl -X POST https://your-domain.com/webhooks/workos \
  -H "Content-Type: application/json" \
  -d '{"test":"data"}' \
  --trace-ascii /dev/stdout | grep "test.*data"
```

### "Webhook endpoint timeout"

**Root cause:** Processing taking >30 seconds, WorkOS times out.

Fixes:
1. Return HTTP 200 BEFORE processing (Step 4D pattern)
2. Move event processing to background queue
3. Check for blocking database queries in request handler

### "Events not appearing in Datadog"

**Root cause:** Integration misconfigured or events not selected.

Fixes:
1. Dashboard → Integrations → Datadog → Verify "Connected" status
2. Check event types are selected in integration config
3. Verify Datadog API key has `logs_write` permission
4. Test with high-frequency event (e.g., trigger SSO login)

### "Missing events in Events API"

**Root cause:** Pagination cursor error or timing issue.

Fixes:
1. Store last processed `event.id` as cursor, not timestamp
2. Check `has_more` field in response before stopping pagination
3. Add overlap buffer: fetch events from 5 minutes before last cursor

### "Duplicate event processing"

**Root cause:** Retry logic without idempotency check.

Fixes:
1. Check `event.id` against processed events table BEFORE processing
2. Use database UNIQUE constraint on event_id column
3. Implement idempotent handlers (upsert instead of insert)

### "Rate limit errors (429)"

**Root cause:** Too many API requests in short window.

Fixes:
1. Check fetched docs for current rate limits
2. Implement exponential backoff on 429 responses
3. Respect `Retry-After` header value
4. Batch event processing instead of per-event API calls
